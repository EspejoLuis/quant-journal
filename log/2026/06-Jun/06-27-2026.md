## 🔑 Key Points

- **Forward declarations break circular header includes:** if a header only uses a type as a reference or pointer in a *declaration*, a forward declaration (`class Instrument;`) is enough — the compiler only needs the name. Full definition moves to the `.cpp`. This is how `engine.hpp` ↔ `instrument.hpp` mutual includes are avoided.
- **Nested types scope, not restrict:** `Engine::Arguments` means the type belongs to the `Engine` namespace, not that only `Engine` can use it. `McEngine::Arguments` inherits from it. `::` is scope resolution, not access control.
- **Object slicing:** passing a derived type by value into a base-type parameter copies only the base part — derived members are cut off. `McEngine::Arguments` passed by value into `Engine::Arguments` loses `payoff`. Passing by pointer avoids all copying; the full derived object is intact on the other side.
- **`dynamic_cast` requires a vtable:** `dynamic_cast` works only on *polymorphic types* — types with at least one virtual function. `Engine::Arguments` has `virtual ~Arguments()`, which gives it a vtable. Without it, `dynamic_cast` is a compile error.
- **`setupArguments(Engine::Arguments*)` — one cast, one engine:** the instrument calls `dynamic_cast<McEngine::Arguments*>(args)`. If the engine passed the wrong arguments type, the cast returns `nullptr` and the instrument throws. No chain of casts — the engine is fixed by `setPricingEngine` so the cast always succeeds for the right engine.
- **QuantLib `instrument.price()` pattern:** pricing flows from the instrument, not the engine. `instrument.price()` calls `setupArguments(engine_->getArguments())`, then `engine_->calculate()`, then returns `engine_->getResult()`. Engine and instrument are paired at construction via `setPricingEngine`.
- **Put-call parity for cash-or-nothing:** $C + P = P_{\text{pay}} e^{-rT}$ (not $K e^{-rT}$). The payout amount $P_{\text{pay}}$ is independent of strike — it is the fixed cash amount received, stored as `payAmount` in `DigitalOptionParameters`.
- **Template full specializations are not inline:** `template <> void BsCloseForm<VanillaEuropeanOption>::calculate()` is a regular function once specialized — defining it in a header included by multiple TUs causes a duplicate-symbol linker error. Fix: declare in `.hpp`, define in `.cpp`.
- **NVI (Non-Virtual Interface):** make the public API non-virtual; make the customisation point `private virtual`. The base class can call the private virtual through virtual dispatch — derived classes override it but callers outside the class cannot invoke it directly.
- **`std::vector<T>{N}` vs `std::vector<T>(N)`:** braces trigger `initializer_list` constructor — `vector<int>{5}` is a 1-element vector with value 5. Parens trigger the size constructor — `vector<int>(5)` is 5 zeros. Use `{}` for literal element values; `()` for a count.
- **`CHECK_THROWS_AS` needs an expression, not a declaration:** `CHECK_THROWS_AS(SomeClass{args}, E)` compiles — a temporary is an expression. `CHECK_THROWS_AS(SomeClass obj{args}, E)` fails — a variable declaration is a statement and cannot appear as a macro argument.

## 🏗️ Architecture Decision — QuantLib-like Engine/Instrument Pattern (revised Jun 27)

Full design in [[engine-instrument-architecture]]. Supersedes the `McEngine::Arguments { payoff std::function }` approach from earlier today.

### Core principle

Instrument drives pricing. Arguments live at the **instrument level**. Engines are **class templates** on the instrument type.

### `Engine` (abstract base)

```cpp
class Engine {
public:
    struct Arguments { virtual ~Arguments() = default; };
    virtual Engine::Arguments* getArguments() = 0;
    virtual void calculate() = 0;
    virtual double getResult() const = 0;
    virtual ~Engine() = default;
};
```

### `Instrument` (concrete base)

```cpp
class Instrument {
public:
    double price();                  // drives the flow — not virtual
    void setPricingEngine(Engine*);
    virtual void setupArguments(Engine::Arguments*) const;  // default throws
    virtual ~Instrument() = default;
private:
    Engine* engine_ = nullptr;
};
```

`price()` flow:

```cpp
Engine::Arguments* args = engine_->getArguments();
setupArguments(args);
engine_->calculate();
return engine_->getResult();
```

### `T::Arguments` (instrument level)

Each instrument defines its own nested Arguments type:

```cpp
struct VanillaEuropeanOption::Arguments : Engine::Arguments {
    OptionParameters params;
    virtual double payoff(double S) const;  // scalar spot → payoff value
};
```

Payoff signature is **scalar `double(double S)`** — mirrors QuantLib's `Payoff::operator()(double)`:

- `McEngine<T>` calls `arguments_.payoff(path.back())`
- `PdeEngine<T>` calls `arguments_.payoff(spaceGrid[i])` for the terminal condition
- `BsCloseFormEngine<T>` reads `arguments_.params` directly for the analytic formula

### `McEngine<T>` (template)

```cpp
template <typename T>
class McEngine : public Engine {
    typename T::Arguments arguments_;   // typed instrument-level member
    ModelParameters modelParams_;
    SimulationParameters simParams_;
    double result_ = 0.0;
public:
    Engine::Arguments* getArguments() override { return &arguments_; }
    void calculate() override;          // simulate paths → arguments_.payoff(path.back())
    double getResult() const override { return result_; }
};
```

### `PdeEngine<T>` (template)

```cpp
template <typename T>
class PdeEngine : public Engine {
    typename T::Arguments arguments_;
    ModelParameters modelParams_;
    PdeParameters pdeParams_;
    double result_ = 0.0;
public:
    Engine::Arguments* getArguments() override { return &arguments_; }
    void calculate() override;          // backward sweep → arguments_.payoff(S_i)
    double getResult() const override { return result_; }
};
```

### `BsCloseFormEngine<T>` (template)

```cpp
template <typename T>
class BsCloseFormEngine : public Engine {
    typename T::Arguments arguments_;
    ModelParameters modelParams_;
    double result_ = 0.0;
public:
    Engine::Arguments* getArguments() override { return &arguments_; }
    void calculate() override;          // reads arguments_.params → BS formula
    double getResult() const override { return result_; }
};
```

Not a subclass of `McEngine<T>` — same engine pattern, no `SimulationParameters`, no path loop.

### `setupArguments` — ONE cast to instrument's own type

```cpp
void VanillaEuropeanOption::setupArguments(Engine::Arguments* args) const {
    auto* a = dynamic_cast<VanillaEuropeanOption::Arguments*>(args);
    if (!a) throw std::logic_error("wrong engine for VanillaEuropeanOption");
    a->params = optParams_;
}
```

Cast always succeeds because `McEngine<VanillaEuropeanOption>`, `PdeEngine<VanillaEuropeanOption>`, and `BsCloseFormEngine<VanillaEuropeanOption>` all store `VanillaEuropeanOption::Arguments arguments_` as a typed member — same QuantLib guarantee.

### Notes

- For path-dependent products (Asian, Barrier, Lookback) `T::Arguments` will add `virtual double pathPayoff(const std::vector<double>& path) const` — scalar `payoff(S)` is not enough. `McEngine<T>` will call `pathPayoff(path)` for those.
- `ChooserEuropeanOption` keeps its dedicated `ChooserEngine` — mid-path BS sub-pricing does not fit the generic template pattern.
- `BsCloseForm` free function (current `blackScholesCloseForm.cpp`) will be replaced by `BsCloseFormEngine<T>` deriving `Engine`.
