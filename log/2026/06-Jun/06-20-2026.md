## 🔑 Key Points

- **Chooser MC pricing — hybrid approach:** simulate paths to $T_c$, compute $C_{BS}(S_{T_c}, K, r, \sigma, T-T_c)$ and $P_{BS}(S_{T_c}, K, r, \sigma, T-T_c)$ analytically at each path, take $\max(C, P)$, discount by $e^{-rT_c}$. No inner MC needed: BS already gives $\mathbb{E}^{\mathbb{Q}}[e^{-r(T-T_c)}\max(S_T - K, 0)\mid S_{T_c}]$ exactly. Outer MC (0 → $T_c$) + inner BS ($T_c$ → $T$ analytic) is strictly better than full nested MC — same outer paths, zero extra variance.
- **Why `payoff(path)` breaks for the chooser:** `McEngine` calls `instrument.payoff(path)`, which receives only the path. At $T_c$ the chooser needs $r$, $\sigma$, and $T - T_c$ to compute BS sub-prices — unavailable in that signature. Solution: dedicated `ChooserMcEngine` with a typed `price(const ChooserEuropeanOption&)` that has direct access to `ModelParameters`. Same pattern as `BsCloseForm`.
- **QuantLib's `Instrument` has no `payoff(path)`:** pricing is entirely engine-driven via `setPricingEngine()` + `NPV()`. Each product fills a product-specific `Arguments` struct; the engine reads it and computes the price. `SimpleChooserOption` derives from `Instrument` (via `Option` → `OneAssetOption`) but does not implement any `payoff(path)` method.
- **Non-pure virtual `payoff()` as the adapter:** making `Instrument::payoff()` non-pure virtual (default: throws) lets `ChooserEuropeanOption` derive from `Instrument` without implementing a path-based payoff. Generic `McEngine` stays intact for vanilla, digital, Asian, barrier; `ChooserMcEngine` uses a typed interface and never calls `payoff()`.
- **Parameter responsibilities for the chooser:** `ModelParameters.timeHorizonInYears` = $T$ (full option expiry, consistent with all other products); `ChooserOptionParameters.choiceDateInYears` = $T_c$ (contractual, stored in the instrument); `ChooserMcEngine` simulates to $T_c$ using `choiceDateInYears`, discounts by $e^{-rT_c}$, and computes remaining BS time as `timeHorizonInYears` − `choiceDateInYears`.
- **Model vs simulation horizon:** $T$ appears in the GBM formula ($S_T = S_0 e^{(r - q - \frac{\sigma^2}{2})T + \sigma\sqrt{T}Z}$) so it belongs in `ModelParameters`. But it is not the simulation horizon for all products — the chooser simulates to $T_c \lt T$. This is why the field was renamed from `maturityInYears` to `timeHorizonInYears`.
- **Products where `payoff(path)` is insufficient — chooser and cliquet only:** all other products (Asian, Barrier, Lookback, Variance Swap, Volatility Swap, Rainbow, Accumulator, Autocallable) can express their payoff purely as a function of the simulated $S$-path, even under Local Vol, Heston, or SLV. The model affects path generation (engine side), not payoff evaluation (instrument side).
