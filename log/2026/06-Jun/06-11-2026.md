## 🔑 Key Points

- **`override` requires a virtual base:** `override` is only valid when overriding a `virtual` method from a base class. Using it in a class with no base (or on a non-matching signature) is a compile error — not just a warning.
- **Object slicing vs pass by reference:** slicing only happens with pass-by-value (derived fields truncated to fit the base layout). With `const Base&`, the full derived object is intact in memory; you just have a restricted view. `dynamic_cast` recovers the derived type at runtime.
- **`const T&` as return type:** a non-const reference cannot bind to a const member. Return `const OptionParameters&` (not `OptionParameters&`) when exposing a const member — otherwise the compiler rejects it, since a non-const reference would allow mutation through the alias.
- **`1/2` is integer division:** `1/2 == 0` in C++. Use `0.5` or `1.0/2.0` for the double literal. Classic trap inside d1's formula.
- **`erfc` over `erf` for `normalCdf`:** for large negative x, `1 + erf(x)` loses precision (cancellation of two nearly equal numbers close to 0). `erfc(-x/sqrt(2))` avoids this — the same result without the subtraction.
- **Analytic engines break the `Engine` interface:** `Engine::price(const Instrument&)` works for MC/PDE because they evaluate any payoff via `payoff(path)`. BS needs the strike directly to compute d1/d2 — forcing it through the interface requires `dynamic_cast`, which is a code smell. Cleaner to not derive `Engine` and take the specific instrument type directly.
